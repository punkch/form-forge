Form Forge for ODK UX review 2026-07-15

Interface critique & enhancement plan

ODK Central integration: from stacked modals to one Central drawer

The feature works. But publishing a form can put three modal layers on screen at once, and the “talk to my server” intent is scattered across three places. Here is what a live pass found — and the direction to fix it.

Method: Interface Craft + agent-browser live pass 8 overlay surfaces reviewed Direction: non-modal drawer · unified hub
01

Context

ODK Central integration is the one networked corner of an otherwise offline, client-only builder. A field-data team uses it to push a form draft to their server and pull published forms back into the library. The emotional context is cautious and infrequent: publishing is deliberate and slightly high-stakes — you’re pushing to a live server other people collect real data on — done by someone who is not thinking about crypto vaults or CORS. They just want their form on their server.

First impression: the individual dialogs are clean, but the flow is built as a stack of overlays that interrupt each other, and the same intent is split across the editor toolbar (Publish), a buried toggle inside the generic Import dialog (Import), and the Settings page (connect & manage). Nothing on screen says “here is your connection to Central and what you can do with it.”

03

What the five lenses found

Ordered by impact: structural (what the interface is) > behavioral (how it feels) > visual (how it looks).

Interface design

Structure & disclosure

Structural · mental model

No home for “my form & its destinations”

A form can be published to several Central destinations (dev / staging / prod), but that list only surfaces inside the Publish dialog — and only as a pre-fill. Close it and every destination is invisible. Publish, Import, and connection management live in three unrelated places.

→One per-form surface listing every destination, plus an add-new action.
Structural · interruption

Vault unlock ambushes the task

Unlock is a session-level ceremony, but it fires mid-task as an interrupting modal the moment you touch a server. The user came to publish and is handed a security prompt on top of their work.

→Resolve unlock inline / up-front — never a modal over a modal.
Behavioral · density

Publish is a mini-app in a 34rem box

One modal carries destination pickers, a new/update radio set, a rename warning, a progress line, a two-branch 409 recovery, an inline error, and a success block with warnings — all self-rearranging in place.

→A legible flow: Destination → Publishing → Result.
Behavioral · feedback

Two confirm patterns, two success patterns

Destructive actions use a ConfirmDialog; the 409 recovery uses inline buttons in a message. Publish success is inline; import success is a toast + navigation. Similar events, inconsistent responses.

→One recovery pattern, one deliberate success pattern.
Consistency & visual

Cohesion & polish

Visual · alignment

Mismatched picker widths

In Publish, the Server select is ~11rem (content-sized) while Project spans the full 34rem. Two controls doing the same job read as unrelated.

→Equal full-width pickers, grouped as one destination.
Visual · guidance

The error names a link that isn’t there

“See the setup guide for how to allow Form Forge to connect” — but there’s no link in the message; the only guide access is a small ? far from the error.

→Make “setup guide” an inline link, right in the error.
Visual · controls

Source toggle isn’t a segmented control

Import’s File / ODK Central switch is two separate buttons (one filled, one outlined). It reads as two actions, not one two-position choice.

→A true segmented control communicates “pick one.”
Behavioral · continuity

No persistent lock / status state

Whether the vault is unlocked — the one thing that governs whether you can publish at all — is only knowable once you trigger an action and get prompted. There’s no glanceable status on the form itself.

→A standing vault-lock state in the rail, plus per-destination freshness.
04

The direction: one Central drawer

Deliver the unified hub through a non-modal side drawer — the same panel language as the properties pane. The form stays visible, nothing stacks, and the form→server relationship finally has a single home. A form can be published to several Central destinations (each a distinct server × project × form — dev, staging, prod), so the drawer tracks every one: each is re-publishable in one click, and “Publish to a new destination” expands inline — a collapsible in the rail, never a modal or a view-swap, so the existing list stays in view. Unlocking the vault is a single inline gate that readies every server at once (one key decrypts them all), not a second modal.

06

How to get there — phased

Phase 1 Low risk

Stop the stacking

  • Inline / up-front vault unlock — Publish & Import never spawn a second modal
  • Equal full-width pickers
  • Inline “setup guide” link in the connection error
  • True segmented source toggle
Phase 2 Medium

The Central drawer

  • Move Publish & Import-from-Central into a non-modal side drawer
  • Destination → Publishing → Result as explicit steps
  • One unified confirm / recovery pattern
  • Form stays visible throughout
Goal Phase 3 Medium

Hub content

  • Standing connection status in the drawer
  • Every destination tracked, with one-click re-publish + add-new
  • Collapse the 3 scattered entry points into one model
  • Import adopts the same drawer language

We’re building the full arc through Phase 3 — the hub shown above. The phases are the delivery order, not a stopping point: Phase 1 removes the “popups on popups” pain on its own, Phase 2 stands the drawer up, and Phase 3 makes it the single home for every destination.

07

Top opportunities, ranked

1
Eliminate modal-on-modal. Never stack the vault prompt over Publish/Import; resolve unlock inline or up-front.
2
Track every destination, in one home. A drawer that owns connection, lists all of this form’s destinations with one-click re-publish, and adds new ones — one place, not three.
3
Make Publish a legible 3-step flow. Destination → Publishing → Result, instead of one dense self-rearranging panel.
4
Fix the visual leaks. Equal picker widths, an inline setup-guide link, a real segmented source toggle.
5
Unify feedback & confirmation. One “are you sure” pattern; one deliberate success pattern (drawer-inline vs toast).
